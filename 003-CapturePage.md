# ADR-003: Capture Page Serving — Pre-built Static HTML on Cloudflare R2 (decoupled from app infrastructure)

**Status:** Accepted
**Date:** March 17, 2026
**Author:** Afterset team
**Affects:** Capture page rendering, fan entry flow (QR/NFC/text-to-join), fan-facing UX, CDN/hosting infrastructure, build pipeline, form submission architecture, offline resilience
**Companion doc:** `capture-page-entry-method.md` — detailed timing chains and browser context analysis for all three entry methods

---

## Decision

**Pre-built static HTML files served from Cloudflare R2 behind Cloudflare's CDN** as the capture page serving infrastructure, fully decoupled from the dashboard application. Each capture page is a self-contained HTML document (~5KB uncompressed, ~3KB after Brotli) generated by a Node.js build script when an artist creates or edits a page, then uploaded to R2 and cache-purged via the Cloudflare API. Form submissions POST to a Cloudflare Worker on the same origin. The entire capture page infrastructure costs **$0/month** through 2M loads/month.

**Critical day-1 design constraints:** System font stack only (no web fonts). CSS gradients only (no background images). All CSS and JS inline in the HTML document. Inline SVG icons for platform links. Dark theme with 7:1+ contrast ratios. Minimum 48×48px touch targets. The page must fit within the **14KB TCP initial congestion window** so the entire document arrives in a single network round trip after TLS completes.

---

## Context

The capture page is the only surface fans ever see. A fan at a live show reaches the capture page via one of three methods — scanning a QR code, tapping an NFC tag, or tapping a link received via text-to-join SMS — and expects a page to appear within 1–2 seconds. If it doesn't, they close the tab and the artist loses them permanently. This is the product's core moment — everything else (dashboard, analytics, email follow-up) is downstream of whether this page loads.

The page architecture is **entry-method-agnostic**: identical HTML is served regardless of how the fan arrives. The entry methods differ in pre-page-load timing (NFC is ~200ms faster than QR; text-to-join adds 8–15 seconds of SMS round-trip before the page load even starts) and browser context (SMS links may open in in-app WebViews with cold caches), but the page load portion — DNS → TLS → HTTP → render — is the same for all three. See `landing-page-entry-method.md` for detailed timing chains per entry method.

### The hostile environment

Indoor music venues present the worst conditions consumer web pages ever face: concrete walls and basements that degrade cellular signal, hundreds of phones competing for limited cell tower bandwidth (effective per-user throughput can drop to 50–100 Kbps), fans standing in dark rooms holding a drink at low screen brightness, and many users on older mid-range Android devices. This is not a theoretical edge case — it's the primary operating environment.

### Scale projections

| Phase | Artists | Capture pages | Page loads/month | Burst pattern |
|---|---|---|---|---|
| Launch (month 1–2) | ~50 | ~200 | ~5,000 | 100–500 hits in 60s per page during a show |
| Traction (month 6) | 200+ | ~1,000 | ~50,000 | Same burst pattern, more concurrent shows |
| Scale (year 1–2) | 10,000 | ~50,000 | ~2,000,000 | 200+ concurrent shows on peak nights |

The 100–500 hits/60s burst describes QR and NFC entry. Text-to-join spreads the same cohort over 2–5 minutes due to SMS round-trip staggering. Both patterns are handled identically by static CDN serving.

### Why this decision carries significant downstream weight

The capture page's serving architecture determines the URL structure (which is baked into every printed QR code, programmed into NFC tags, and included in SMS reply messages), the build pipeline (how artist edits propagate to the live page), the form submission path (which affects the queue architecture from ADR-001), and whether the page can work offline in signal-dead venues. Changing the URL structure after QR codes are printed on posters and merch is impossible. Changing the serving infrastructure after launch is feasible but requires careful CDN migration to avoid cache misses during transition.

**URL length constraint across entry methods:** QR codes support ~2,900 characters (no pressure). NFC NTAG213 tags (cheapest, most common) have ~137 usable bytes — the URL must stay under ~120 characters. SMS reply messages share 160 characters with the message text — shorter URLs leave more room for the message. The scheme `afterset.net/c/[slug]` at ~40 characters fits all three comfortably. Capture page URLs should stay under 50 characters total, including any query parameters.

### The physics constraint

Research revealed an irreducible network overhead floor on slow 3G (300 Kbps, 400ms RTT): DNS resolution (400ms) + TCP handshake (400ms) + TLS 1.3 handshake (400ms) = **1,200ms before a single byte of HTML arrives**. This means the sub-1-second FCP target is physically impossible on a fully cold first request regardless of architecture. The rendering strategy choice only controls ~600ms of the 2-second budget. Optimizing the network chain (same-origin serving, direct URLs in all entry methods, HTTP/3) matters more than any server-side decision. [VERIFIED — protocol specifications]

### Constraints

- Solo developer — operational simplicity is critical.
- Page weight must be under 50KB total; target under 14KB (TCP initial congestion window) for single-round-trip delivery.
- Page must load in under 2 seconds on slow 3G (300 Kbps, 400ms RTT).
- Must handle burst traffic (500 concurrent requests to one page) with zero cold start risk.
- Dark theme with high contrast is non-negotiable (fans at 10–30% screen brightness).
- Must work on mid-range Android devices (Moto G, Samsung Galaxy A series).
- Form submission must be resilient to connectivity loss mid-submission.
- Capture page infrastructure cost must be negligible — ideally $0 at launch.

## Options Considered

### Option A: Pre-built Static HTML on Cloudflare R2 + CDN — SELECTED

**What it is:** Each capture page is a complete, self-contained HTML file (~5KB) generated at edit time by a Node.js build script. Files are stored in Cloudflare R2 (S3-compatible object storage) and served through Cloudflare's CDN at 330+ edge locations. Form submissions go to a Cloudflare Worker on the same origin. No framework, no runtime, no server-side rendering.

**Request lifecycle:** Fan reaches URL (via QR scan, NFC tap, or SMS link tap) → browser requests `afterset.net/c/artist-slug` → Cloudflare CDN edge cache hit (or R2 origin fetch on first request, ~50–150ms) → complete HTML document returned → browser parses and renders inline CSS/JS → page visible. Total round trips: DNS (1) + TCP (1) + TLS 1.3 (1) + HTTP request/response (1) = 4 round trips. The entire page arrives in the HTTP response — no subsequent requests for CSS, JS, fonts, or images.

**Performance:** ~5ms TTFB from CDN edge cache. Total load on cold slow 3G: **~1,818ms**. With HTTP/3 (saves 1 RTT) and cached DNS (saves 1 RTT): **~1,000ms**. During a concert burst, after the first few fans load, DNS caching at the cellular carrier and Cloudflare's edge cache reduce subsequent loads further. [INFERRED from protocol math and CDN behavior]

**Cost:** $0/month. R2 free tier includes 10M Class B reads/month and zero egress fees. 50K HTML files at ~5KB each = ~250MB storage (well within R2's 10GB free tier). The Cloudflare Worker for form submission uses the free tier (100K requests/day). [VERIFIED — Cloudflare R2 pricing]

**Edit-to-live latency:** Artist saves in dashboard → webhook triggers build script → script generates HTML from template + artist data → uploads to R2 via S3 API → purges CDN cache via Cloudflare API → live in **5–10 seconds**. [INFERRED]

**Cold start risk:** Zero. Static files on a CDN have no compute layer to cold-start. The worst case is a CDN edge cache miss requiring an origin fetch from R2, adding 50–150ms — invisible to the user. [INFERRED from standard CDN behavior]

**Burst handling:** CDN-native. 500 concurrent requests to the same URL hit the edge cache and return instantly. No connection pooling, no function invocation limits, no serverless concurrency caps. [VERIFIED — CDN architecture fundamentals]

**Risks:**

1. **Build pipeline complexity.** A custom Node.js script must generate HTML files, upload to R2, and purge the CDN cache. This is ~100 lines of code but is custom infrastructure that must be maintained. If the build script fails silently, a fan sees a stale page.
2. **No dynamic content injection.** The page is static — anything that needs to change at request time (geo-personalized content, A/B test variants, real-time attendance counts) requires migrating to Workers + KV.
3. **R2 has no automatic `index.html` resolution.** Requests must use explicit paths (`/c/artist-slug`, not `/c/artist-slug/`). Cloudflare Workers can add this, but that moves into the Workers + KV option.

---

### Option B: Cloudflare Workers + KV — PLANNED UPGRADE

**What it is:** A Cloudflare Worker reads artist data from KV (key-value store) at the edge, injects it into an HTML template string, and returns the complete page. No origin server — the Worker is the origin.

**Performance:** 10–30ms TTFB (Worker execution + KV read). Total load on cold slow 3G: **~1,830–1,860ms**. Workers claim "zero cold starts" by preloading isolates during the TLS handshake, though community reports show 50–250ms first-request latency for idle Workers. [VERIFIED — Cloudflare blog; REPORTED — community forums]

**Cost:** $5/month (Workers Paid plan). KV reads are included. [VERIFIED — Cloudflare pricing]

**Key advantage over Option A:** Dynamic content at the edge. A/B testing, geo-personalized messaging, real-time analytics injection, and request-time logic become trivial. Artist edits propagate via KV write instead of file upload + cache purge.

**Key disadvantage:** KV writes are eventually consistent — **propagation takes up to 60 seconds globally**. An artist saves a page edit and a fan on the other side of the world might see the old version for up to a minute. For Afterset's use case (artists edit pages hours before a show, not during), this is acceptable. [VERIFIED — Cloudflare KV documentation]

**Why not selected for launch:** Unnecessary complexity for v1. The capture page has no dynamic content requirements at launch — artist data is known at edit time and doesn't change at request time. The Worker adds a moving part (template logic, KV reads, Worker deployment) with no user-facing benefit over pre-built static files. Option A is strictly simpler with identical performance.

**When to migrate:** When any of these features are needed: A/B testing capture page variants, geo-targeted value exchange messaging, real-time show attendance display, or per-request analytics. Estimated: traction phase (~200 artists). Migration is minimal: move the HTML template into the Worker, write artist data to KV instead of generating HTML files, update DNS routing.

---

### Option C: Bunny CDN + Storage — STRONG ALTERNATIVE

**What it is:** Same static file approach as Option A but on Bunny's CDN infrastructure instead of Cloudflare.

**Performance:** Bunny consistently ranks #1 on CDNPerf benchmarks with sub-25ms average global latency across 119 PoPs. Edge Storage replication gives ~41ms first-byte. [REPORTED — bunny.net, CDNPerf]

**Cost:** ~$1–2/month ($0.01/GB bandwidth, $1/month minimum). [VERIFIED — bunny.net pricing]

**Why not selected:** Afterset's DNS is already on Cloudflare (landing page at afterset.net). Using Cloudflare R2 means the CDN, DNS, SSL, cache purging, and Workers (for form submission and later upgrades) all live in one account with one API. Bunny would require configuring a separate CDN provider, managing SSL certificates separately, and losing the seamless Workers upgrade path. The ~20ms TTFB difference between Cloudflare and Bunny is invisible to users given the 1,200ms network overhead floor. Bunny is the recommended alternative if Cloudflare R2 proves unreliable or introduces egress fees.

---

### Option D: Next.js ISR on Vercel — NOT RECOMMENDED

**What it is:** Capture pages as ISR routes within the main Next.js application. Pages are statically generated on first request and cached, with on-demand revalidation when artists edit.

**Performance on cache hit:** ~5ms TTFB, ~1,818ms total. Comparable to static CDN. On cache miss (the critical scenario): 200–400ms TTFB from serverless function execution, pushing total load to **~2,100ms — exceeding the 2-second target**. [REPORTED — Vercel benchmarks]

**The fatal flaw: cold starts after idle periods.** Afterset's traffic pattern is long stretches of zero traffic (days between shows) followed by sudden bursts. After days of inactivity, ISR cache misses trigger serverless function cold starts of **250ms–1,500ms at P50, potentially 2–7 seconds in worst cases**. The first fan at a show — the most important fan, the one who proves the product works — gets the worst experience. [REPORTED — OpenStatus benchmarks, GitHub discussions]

**Cost:** $20–21/month (Vercel Pro, which is needed for the dashboard anyway). [VERIFIED]

**Why not recommended:** Cold start risk directly contradicts the product requirement. The burst pattern (zero → 500 requests in 60 seconds) is the exact scenario where serverless cold starts are worst. At 4× the cost of Cloudflare with worse cold-start behavior for this specific traffic pattern, ISR only makes sense if the capture page requires Next.js-specific features (it doesn't — it's a static HTML form).

**Mitigation if forced onto Vercel:** Set extremely long ISR `revalidate` times (24 hours+), use on-demand revalidation only, and implement a pre-warm cron job that hits every active capture page URL daily. This reduces cold start probability but doesn't eliminate it and adds operational complexity.

---

### Option E: SSR on Origin Server — NOT RECOMMENDED

**What it is:** Server-side rendered HTML generated per request from a Node.js/Next.js server.

**Why not recommended in one sentence:** Adds 200–800ms of server processing time to every request, pushing slow 3G load times to 2.1–2.6 seconds, with the worst cold start risk of any option and no caching benefit for a page that changes only when the artist edits it. [INFERRED]

---

### Option F: Vercel Edge Functions — VIABLE BUT UNNECESSARY

**What it is:** Capture pages rendered by Vercel's edge runtime within the Next.js app. Similar concept to Cloudflare Workers but within the Vercel ecosystem.

**Performance:** 20–50ms TTFB, ~1,830–1,860ms total. Low cold start risk. [REPORTED]

**Cost:** $20–22/month (requires Vercel Pro). [VERIFIED]

**Why not selected:** Same performance as Cloudflare Workers at 4× the cost. Doesn't offer the seamless upgrade path to Workers + KV. Only makes sense if the team decides to consolidate everything on Vercel for operational simplicity — but this ADR's finding is that decoupling the capture page from the dashboard infrastructure is the right architectural choice.

## The Architecture That Makes This Work

### Payload budget: everything fits in one TCP round trip

The TCP initial congestion window is 14KB — any document under this threshold arrives in a single network segment after TTFB. Research confirmed that a complete, professional capture page compresses to **~2.8–3.2KB via Brotli**, leaving ~11KB of headroom. [INFERRED from prototype specification]

| Component | Uncompressed | Notes |
|---|---|---|
| HTML structure + meta tags | ~750B | Semantic, accessible, minimal; includes OG tags for SMS link previews |
| Inline CSS (`<style>`) | ~1,000B | Gradient bg, card layout, form, buttons, responsive |
| Inline JS (`<script>`) | ~630B | `fetch()` submit + localStorage offline queue + entry method param read |
| Inline SVG icons (7 platforms) | ~2,800B | ~300–500B per optimized brand icon via SVGO |
| Template variables (artist data) | ~400B | Name, URLs, colors, value exchange text |
| **Total uncompressed** | **~5,580B** | |
| **After Brotli 11** | **~2,900–3,300B** | Pre-compressed at build time |

**OG meta tags (~150B) are required for SMS link previews.** When a capture page URL is sent via iMessage, the recipient's device fetches it to generate a preview. Without OG tags, the preview is a bare URL. With `og:title` (artist name) and `og:description` (value exchange message), the preview looks intentional. These tags also cause bot pageviews from Apple's link preview fetcher — server-side analytics should filter bot user agents to avoid inflated counts.

**Entry method tracking (~80B JS) enables per-method analytics.** A one-liner reads the `?v=` query param and populates a hidden form field: `q` = QR, `n` = NFC, `s` = SMS, `d` = direct/unknown. This feeds into the per-gig analytics dashboard's capture method breakdown without any server-side detection logic.

**What this eliminates:** Web fonts (system font stack: `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif` — Roboto is pre-installed on every Android device). Background images (CSS `linear-gradient()` costs ~60 bytes). External CSS/JS files (everything inline). Third-party scripts (analytics tracked server-side via the form submission API). Framework runtime (no React, no Next.js client bundle). [VERIFIED for font availability; INFERRED for byte counts]

### Same-origin serving saves 400ms

DNS resolution costs 400ms+ per unique domain on slow 3G. If the capture page and form submission API share the same origin (`afterset.net`), fans only pay the DNS cost once. Serving capture pages from `afterset.net/c/[slug]` and accepting form POSTs at `afterset.net/api/capture` — both through Cloudflare — eliminates a second DNS lookup entirely. If the fan-capture API must live on a different origin, a `<link rel="preconnect">` tag in `<head>` parallelizes the second DNS+TCP+TLS with page rendering. [VERIFIED — networking fundamentals]

### All entry methods must use direct URLs — never shorteners

A URL shortener (bit.ly, etc.) adds a redirect costing 800ms+ on slow 3G: DNS for the shortener domain (400ms) + redirect RTT (400ms). This applies equally to QR codes, NFC tags, and SMS reply links. All must encode the final URL directly (`https://afterset.net/c/artist-slug`). QR codes support ~2,900 characters; NFC NTAG213 tags support ~120 characters; SMS messages share 160 characters with the reply text — the ~40-character canonical URL fits all three. For SMS, keeping the URL short organically is better than shortening it: `afterset.net/c/jdoe` leaves 120+ characters for the reply message, which is plenty.

This constraint must be enforced in the QR code generation feature, the NFC programming instructions, and the text-to-join reply template: never allow shortener URLs in any entry method. If entry method tracking is needed, use minimal query params (`?v=q`, `?v=n`, `?v=s` — 4 characters) rather than UTM tags. [INFERRED — protocol math]

### Offline-resilient form submission via localStorage

Venue connectivity can drop between page load and form submission. The capture page uses a **localStorage-first submission pattern**:

1. Fan submits form → JS intercepts with `preventDefault()`
2. Form data serialized to `localStorage` immediately (before any network call)
3. Success UI shown instantly ("You're in!") — fan sees confirmation before network responds
4. `fetch()` POST fires with `keepalive: true` and a 10-second `AbortController` timeout
5. On successful POST, `localStorage` entry cleared
6. On page load, check `localStorage` for unsent submissions and retry

This is more reliable than Service Workers for first-visit fans because it works immediately without SW registration delay and covers all browsers. Service Workers have a race condition on first visit: if connectivity drops after HTML loads but before `sw.js` downloads, the SW never installs. Background Sync API provides automatic retry but is Chromium-only (Chrome for Android, Samsung Internet, Edge — not Safari or Firefox). For the target audience (mid-range Android at concerts), Chromium coverage is sufficient for progressive enhancement, but localStorage is the reliable baseline. [VERIFIED — caniuse, W3C spec]

### Dark theme with venue-optimized contrast

Fans hold phones at 10–30% brightness in dark venues. The capture page enforces: pure black (#000000) background, white (#FFFFFF) text, high-saturation accent colors for CTA buttons (carrying forward the honey-gold #E8C547 from the landing page design system), minimum 7:1 contrast ratio (exceeding WCAG AA's 4.5:1 requirement), minimum 48×48px touch targets, minimum 16px body text / 20px+ form labels. No subtle grays — they vanish on dim screens. [VERIFIED — WCAG specification, Material Design guidelines]

### Browser context varies by entry method

QR scans and NFC taps open the device's default browser (Safari on iOS, Chrome on Android). SMS links, however, may open in in-app WebViews if the fan uses WhatsApp, Telegram, or Facebook Messenger instead of the default Messages app. In-app WebViews have cold DNS caches, no HTTP/3 0-RTT connection reuse, and no shared cookie/cache state with the system browser — they represent the worst-case browser context. The inline-everything approach (no external CSS, JS, fonts, or images) is already optimized for this scenario: there are zero additional resources to fetch regardless of browser state.

### Capture API must deduplicate across entry methods

If an artist uses QR + NFC + text-to-join at the same show, a fan might reach the capture page via multiple methods. The `/api/capture` endpoint must **upsert on `(artist_id, email)`** rather than insert. Return 200 OK regardless — the fan sees "You're in!" even on a duplicate submission. Track the entry method (`v` query param) for analytics but don't create duplicate fan records. This is an API design requirement, not a page architecture concern, but it must be designed from the start.

### Build pipeline

```
Artist edits page in dashboard
  → API saves to Supabase
  → Webhook triggers build script (Node.js, ~100 lines)
  → Script reads artist data from database
  → Generates complete HTML from template + data
  → Pre-compresses with Brotli level 11
  → Uploads to R2 via S3-compatible API
  → Purges Cloudflare CDN cache for that URL
  → Live in 5–10 seconds
```

The build script is the single custom component in this architecture. It must be monitored — if it fails silently, fans see stale pages. Implement: retry on upload failure, dead-letter alerting if a page hasn't been updated after 3 retries, and a manual "rebuild all pages" endpoint for recovery.

## Upgrade Path

| Phase | Architecture | Monthly cost | Migration effort |
|---|---|---|---|
| Launch (50 artists) | Static HTML on R2 + CDN, Worker for form POST | $0 | — |
| Traction (200+ artists) | Add Workers + KV when A/B testing or dynamic features needed | $5 | Move template to Worker, write data to KV instead of generating files. ~1 day. |
| Scale (10K artists) | Workers + KV with custom domain support per artist | $5–15 | Add domain routing in Worker. Build pipeline complexity increases. |

The critical architectural property: **the URL structure (`afterset.net/c/[slug]`) remains stable across all phases.** QR codes printed during the launch phase, NFC tags programmed during setup, and text-to-join keywords configured in Twilio all continue to work at every subsequent phase. The serving infrastructure behind the URL changes; the URL does not.

## Consequences

### Positive

- Zero-cost capture page serving through 2M loads/month — capture infrastructure is free.
- Zero cold start risk — static CDN files have no compute layer to cold-start. The first fan at a show gets the same performance as the hundredth.
- Entire page fits in a single TCP round trip (~3KB compressed), meaning fans receive the complete rendered page in one network exchange after TLS.
- Fully decoupled from the dashboard application — the capture page serving can't be affected by dashboard deployments, Vercel outages, or application bugs.
- HTTP/3 and TLS 1.3 enabled by default on Cloudflare — saves 400ms per connection on the protocols alone.
- localStorage-first submission pattern means fans see success confirmation even if connectivity drops mid-submit.
- No framework runtime on the fan-facing page — zero JS bundle size, zero hydration time, zero client-side routing overhead.
- Pre-compressed with Brotli 11 at build time — zero runtime compression CPU cost.

### Negative

- Custom build pipeline (~100 lines of Node.js) that must be built, tested, and monitored. If it fails silently, fans see stale pages.
- No dynamic content at request time — A/B testing, geo-personalization, and real-time data injection require migrating to Workers + KV.
- System font stack means the capture page doesn't use Afterset's brand fonts (Bricolage Grotesque / DM Sans from the landing page). The capture page will look slightly different from the marketing site. This is an intentional trade-off: brand consistency costs 16–20KB per font weight, which is 5–7× the entire page weight.
- Separate infrastructure from the dashboard means two deployment targets, two monitoring surfaces, and two sets of credentials to manage (Cloudflare for capture pages, Vercel/Railway for dashboard).
- The R2 + CDN architecture has no built-in admin UI for debugging — checking what HTML a specific page is serving requires an API call or R2 dashboard inspection, not a database query.

### Mitigations

1. **Monitor the build pipeline.** Log every build-and-upload cycle. Alert if any page fails to upload after 3 retries. Implement a "rebuild all" endpoint for disaster recovery. Include a build timestamp in the HTML as a comment (`<!-- built: 2026-03-17T14:30:00Z -->`) so stale pages are identifiable.

2. **Test on real hardware at a real venue.** Before launch, load the capture page on a Moto G (2020 or older) and a Samsung Galaxy A series at a live music venue during soundcheck. Measure actual load time. If it exceeds 3 seconds scan-to-render, investigate the specific bottleneck (DNS? TLS? RF congestion?).

3. **Pre-warm DNS for the first show.** When an artist activates a capture page for a gig, send a preflight request from the build pipeline to prime the Cloudflare edge cache at the nearest PoP. This ensures the first fan's request hits a warm edge cache, not an R2 origin fetch.

4. **Validate the offline submission path.** Load the page on a throttled connection, disable the network adapter, submit the form, re-enable the network, and verify the submission reaches the API. Test on both Chrome for Android and Safari iOS. This must be a repeatable QA step before every capture page template change.

## Revisit When

- **Cloudflare R2 introduces egress fees or changes free tier limits.** Evaluate Bunny CDN + Storage as a drop-in replacement (~$1–2/month for equivalent functionality).
- **A/B testing or geo-personalization is needed on capture pages.** Migrate to Workers + KV (~1 day of work, $5/month). The decision trigger is the first product requirement that needs request-time logic.
- **Artist demand for custom domains on capture pages emerges.** Custom domains (`fans.bandname.com`) require Cloudflare for SaaS (CF4SaaS) or a multi-domain SSL solution. This is a significant complexity increase — evaluate at 200+ artists when the feature request pattern is clear.
- **The dashboard consolidates on a platform with excellent edge rendering.** If the app moves to a platform where edge-rendered capture pages are trivially easy (e.g., Cloudflare Pages with Functions), the static file approach may be unnecessarily complex by comparison. Re-evaluate infrastructure consolidation at that point.
- **Page weight exceeds 14KB due to feature additions.** If new requirements (background images, custom fonts, embedded audio previews) push the page past the TCP initial congestion window, the single-round-trip advantage is lost. Reassess the payload budget and consider lazy-loading non-critical elements.
- **Real-venue testing shows load times exceeding 3 seconds.** Investigate whether the bottleneck is network (reduce page weight further, add service worker for repeat visits) or infrastructure (evaluate Bunny CDN's faster edge network). The 3-second threshold is the "fan gives up" boundary based on industry mobile abandonment data.

## Validation Tasks Before Committing

These should be completed during the Pre-Build Phase (Day 3–4 of the build roadmap, alongside ADR-001 and ADR-002 validation):

1. **[ ] Build the prototype capture page.** Implement the HTML template from the research prototype specification. Measure uncompressed size (target: <6KB) and Brotli-compressed size (target: <4KB). Verify it fits within 14KB TCP initial congestion window.

2. **[ ] Test on WebPageTest with slow 3G profile.** Run the prototype through WebPageTest: Moto G Power, Chrome, slow 3G (300 Kbps, 400ms RTT). Pass criteria: total load <2,000ms, FCP <1,000ms on warm DNS, total transfer <14KB.

3. **[ ] Upload to R2 and test the full CDN path.** Create a Cloudflare R2 bucket, upload the prototype, configure CDN caching, and test from multiple geographic locations via WebPageTest. Measure TTFB from edge cache (target: <50ms).

4. **[ ] Test burst traffic.** Run k6 or Artillery with 500 concurrent requests to a single capture page URL on R2. Verify all requests return HTTP 200 with <100ms TTFB.

5. **[ ] Test offline form submission.** Load the page on a throttled connection, disable the network, submit the form, re-enable the network, verify the submission reaches the API endpoint. Test on Chrome for Android and Safari iOS.

6. **[ ] Test at a real venue.** Load the capture page on a mid-range Android phone at a bar or music venue. Measure actual scan-to-render time with a stopwatch. If no venue is available, test from a basement or concrete building with poor cellular signal.

7. **[ ] Validate build pipeline.** Write the build script, generate 10 test pages, upload to R2, purge cache, and verify all pages are live and current within 15 seconds.

8. **[ ] Test dark-theme legibility.** Screenshot the capture page at 10% screen brightness on a phone. Verify all text, the email input, and the submit button are clearly visible.
